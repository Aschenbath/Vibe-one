# Vibe-one Product Studio 与成品质量流水线设计

日期：2026-07-11

## 1. 背景与问题

Vibe-one 已具备完整的文字/截图输入、多模态规划、React + Vite 生成、构建与交互验收、本地视觉比较、有界修复和证据回放能力，但当前完成度仍停留在“功能可用”：

- Product Lab 由历史栏、事件卡、证据卡和大面积空白组成，信息层级偏平，缺少真正的作品画布与质量控制中心。
- 当前 builder 主要约束“能运行、能交互”，8 文件与约 12,000 字符预算迫使模型把页面、状态和样式压缩在少量文件中，容易产生模板化或半成品 UI。
- 无参考图任务没有视觉相似度门禁；机械 reviewer 能证明功能正确，但不能证明界面具备成熟产品所需的结构、响应式和状态完成度。
- 现有 Expense Tracker 与 Notes App 演示规模较小，不能代表数据密集型 B2B 工作流或 AI 原生知识产品的生成能力。

本轮目标不是继续堆功能，而是把工作台和生成结果同时提升到可作为核心作品展示的成熟度。

## 2. 目标

1. 将 Product Lab 升级为桌面优先的 `Product Studio`：清晰区分任务输入、制作过程、作品画布、质量检查和交付证据。
2. 将生成流程升级为“产品任务书 -> 设计规格 -> 初稿 -> 功能验收 -> UI 质量验收 -> 有界抛光 -> 最终交付”。
3. 让纯文字任务在没有参考图时也必须满足可复现的 UI 完成度门槛。
4. 以两个高价值演示替换 README 主展示中的低复杂度演示：数据密集型 B2B SaaS 与 AI 原生知识工作台。
5. 保持既有安全边界：固定脚手架、依赖白名单、路径 jail、机械成功门槛、秘密脱敏和有界模型调用均不得回退。

## 3. 已选择方案

采用 `Product Studio + 成品质量流水线`，视觉方向为“Linear 的精密感 + Figma/Framer 的作品舞台”。

拒绝以下方案：

- 仅替换 CSS 和演示：无法改善生成器的成品质量。
- 引入 design agent / builder agent / critic agent 多代理编排：成本、时延和不确定性过高，也会削弱当前单 provider、机械验收的清晰架构。

本轮仍使用一个 OpenAI-compatible provider。模型负责规划、生成和按证据修整；是否成功始终由本地机械检查决定，禁止模型自评。

## 4. Product Studio 信息架构

### 4.1 Focus：产品任务书

新建任务不再表现为普通表单卡片，而是一份集中式产品任务书：

- 产品目标：用户要完成什么、最终交付是什么。
- 目标用户：主要角色、使用环境和信息密度。
- 核心流程：最多 3 个必须完成的用户任务。
- 视觉方向：气质、色彩倾向、密度和参考产品描述。
- 参考截图 storyboard：缩略图、页面角色、顺序、删除和拖放。
- 高价值起点：B2B 工作台、数据产品、AI 知识工具；只提供少量结构化起点，不建设模板市场。

兼容现有 `brief` 输入：前端把结构化字段组合为清晰的 brief，服务端和 CLI 仍接收统一配置，不引入新的远程数据模型。

### 4.2 Flow：三段式制作台

运行后切换为三段式 Studio：

1. **生产时间线（左）**
   - 阶段：设计规格、初稿生成、构建、交互验收、UI 质量验收、视觉验收、成品抛光、交付。
   - 默认显示当前动作、耗时、通过/失败状态；展开后显示原始事件和修复轮次。
   - 不再让事件日志长期占据半个主画面。

2. **作品画布（中）**
   - 占据最大可用面积，始终展示当前产物、构建状态或明确的空状态。
   - 支持桌面、平板、手机视口切换；刷新、重新启动和新窗口打开。
   - 页面切换与设备切换不重新启动任务。
   - 运行期间显示最近一次可用构建；新构建完成后无跳动替换。

3. **Inspector（右）**
   - 产品规格：目标用户、页面、场景和 acceptance criteria。
   - 设计系统：tokens、字体层级、间距、组件和响应式规则。
   - 质量：构建、交互、UI 质量、视觉比较与总状态。
   - 证据：参考图、结果图、各轮比较、修复记录和交付报告。
   - 每个失败项明确关联页面、检查名和证据，不显示无上下文的错误字符串。

### 4.3 窄屏

- 作品画布保持主内容；时间线与 Inspector 收入左右抽屉。
- 顶部保留任务状态、设备切换和核心操作。
- 390px 只保证查看、切换证据和启动预览，不重新设计为移动端生产工具。

## 5. 视觉系统

- 背景：浅瓷色或低饱和冷灰，不使用普通后台式纯灰大底。
- 文字：深石墨色；标题强调清晰的编辑感，不使用夸张营销大字。
- 主操作：钴蓝；成功使用克制的青绿色；警告使用琥珀；失败使用低饱和红。
- 形态：减少大圆角卡片堆叠，更多使用细分隔线、画布框、分区标题和紧凑工具条。
- 图标：统一描边图标，不使用 Emoji 充当产品图标。
- 动效：只用于画布切换、抽屉、阶段推进和证据替换，180–240ms，支持 reduced motion。
- 字体：使用无需联网的本地中文字体栈；通过字重、字距和层级建立品牌感，不依赖远程字体。
- 所有交互目标至少 44px，并覆盖 hover、focus-visible、pressed、disabled、loading、empty、error、success。

## 6. 产品任务书与设计规格

Planner 输出在现有字段上增加 `productDesign`，包括产品类型、目标用户、视觉语气、信息密度、导航、内容策略、设计 tokens、组件语言、核心状态和响应式规则。

约束：

- 字段必须与产品类型、页面结构和用户任务一致，不能输出空泛的“现代、简洁”。
- 颜色、字体、间距和圆角必须形成可执行 tokens。
- `requiredStates` 只要求与核心流程相关的状态，不为了凑数生成无意义页面。
- 有参考图时，`productDesign` 与既有 `visualDesign`、页面映射共同构成批准规格。

## 7. 初稿生成合同

- 初稿上限从 8 文件 / 约 12,000 字符调整为 12 文件 / 约 24,000 字符，继续使用 streaming 与 delimiter 协议。
- 每个规划页面有明确组件边界；layout、tokens 和通用组件可复用，但禁止拆空壳组件。
- 必须定义并实际使用 CSS variables/design tokens。
- 使用现实且一致的 mock data，禁止 lorem ipsum、`Card 1`、`Item A` 等占位内容。
- 实现规格要求的 loading/empty/error/success 状态及触发方式。
- 禁止使用 Emoji 充当功能图标；允许将 `lucide-react` 加入固定白名单，不加入图表库，简单图表使用 CSS/SVG。
- 不允许交付未样式化的浏览器原生按钮、输入框、select 或 number spinner。

## 8. UI 质量验收

新增本地确定性的 `UI quality audit`，不调用模型评分。每个核心页面在桌面和移动 viewport 检查：

- 文档与主容器无横向溢出，关键元素不越界或重叠。
- 按钮、链接和表单控件的可点击区域不小于 44px。
- 文本对比度满足 WCAG AA（普通文本 4.5:1，大文本 3:1）。
- 页面存在明确的 main/navigation/heading 层级。
- 核心状态可通过 fixture 或交互场景进入，并显示对应文本。
- 没有空白主区域、TODO、异常堆栈、无样式原生控件、用户可见 Emoji 图标或禁用占位词。
- 桌面与移动截图非空，并记录页面尺寸和检查结果。

UI audit 只检查可复现的完成度信号，不用规则假装判断“好看”。失败项进入既有有界 repair loop。

## 9. 成品抛光阶段

初稿首次通过功能和 UI 质量检查后，完整运行模式执行一次 `polish pass`：

- 输入批准规格、当前源码、桌面/移动截图、UI audit、参考图和视觉比较（如有）。
- 只改善层级、排版、密度、状态表现、组件一致性和响应式，不添加新功能。
- 最多返回 4 个完整文件、约 18,000 字符，继续受 safeJoin、禁写清单和固定依赖约束。
- `maxPolishRounds` 固定为 1；Plan Only 不执行 polish。
- polish 在候选副本上应用。候选重新通过构建、内容、交互、UI audit 和视觉门禁后才能成为最终 app。
- 候选失败时以 `POLISH_FAILED` 结束并保留初稿/候选证据，禁止静默交付未通过版本。

Polisher 可以查看截图并修改代码，但不能给自己打分；最终结论仍来自本地 reviewer。

## 10. 流水线、事件与错误

```text
任务书/截图 -> 产品与设计规格 -> 初稿 -> 功能验收 -> UI audit
-> 参考图视觉门禁（如有） -> 有界修复至首次全绿
-> 单次 polish candidate -> 全量复验 -> 证据与交付报告
```

新增事件：`design:done`、`quality:audit`、`polish:start`、`polish:applied`、`polish:failed`。

UI 阶段文案：设计规格、生成初稿、功能验收、质量检查、成品抛光、交付完成。

新增稳定错误码：`PRODUCT_DESIGN_INVALID`、`UI_QUALITY_FAILED`、`POLISH_FAILED`。公开事件和对象继续只暴露稳定 code 与安全中文文案。

## 11. 代表性演示

### 11.1 SignalDesk：AI 客服质检与运营平台

输入模式：纯文字任务书，证明没有参考图也能生成成熟界面。

页面：运营总览、会话队列、质检详情。内容包含 SLA、满意度、风险会话、趋势、团队负载、规则命中、评分拆解、证据和处置动作。

至少验证：筛选风险会话、搜索、打开详情、分配负责人、标记复核、返回队列。

### 11.2 Atlas Research：研究情报工作台

输入模式：任务书 + 至少 2 张参考图，证明多模态页面映射和视觉修复。

页面：资料库、阅读工作区、洞察集合。内容包含来源筛选、标签、双栏阅读、引用证据、洞察卡片、搜索和收藏。

至少验证：搜索资料、打开阅读、切换引用、保存洞察、筛选集合、返回资料库。

### 11.3 演示证据

- 两个主演示必须通过真实 OpenAI-compatible API 运行生成；手工编写或 stub 产物只能用于回归测试，不能作为 README 主证据。
- 每个主演示至少 3 个真实页面、5 个交互场景、桌面/移动截图和 Delivery Report。
- Atlas 额外包含参考图映射、各轮视觉分数和视觉修复证据。
- 公开 mock data 使用中性产品数据，不包含个人信息、私有 endpoint 或 API Key。
- Expense Tracker 与 Notes App 保留在历史归档，但从 README 主展示移除。

## 12. 测试策略

### 单元与集成

- Planner 校验 `productDesign` 的必填字段、tokens 和状态一致性。
- Builder prompt、预算与依赖白名单合同。
- UI audit 的 overflow、重叠、hit target、contrast、状态和占位词检查。
- Polisher 输入包含源码、桌面/移动截图和失败证据，输出仍受文件边界限制。
- polish candidate 只有全量复验通过后才被提升为最终 app。
- 旧 brief-only、reference-only、combined 和安全测试保持通过。

### 浏览器工作台

- Focus 产品任务书、storyboard、结构化起点和设置操作。
- Flow 三段布局、设备切换、时间线/Inspector 抽屉和阶段推进。
- design spec、quality failures、polish evidence 和历史回放。
- 1440×900 主要视觉验收；390×844 验证核心查看流程和无横向溢出。
- reduced motion、键盘焦点和 44px 目标。

### Pipeline E2E

- 无参考图但 UI audit 失败 -> repair -> 全绿 -> polish -> 最终全绿。
- polish candidate 破坏交互 -> `POLISH_FAILED`，不得标记成功。
- combined input 视觉失败 -> repair -> polish -> 最终视觉与功能全绿。
- 两个代表演示保存经过人工检查的截图和报告。

## 13. 文档与展示

- README 首屏改为 Product Studio 与两个代表性演示，不再把记账/笔记作为核心能力证明。
- 提供一张 Focus、一张 Studio Flow 和两个演示的代表截图。
- 文档明确区分机械 UI 完成度、参考图粗粒度相似度和人工视觉验收，禁止包装为像素级设计质量。
- `history.md` 记录真实模型、运行轮次、场景数量、UI audit 和剩余边界；不记录 Key 或私有 endpoint。

## 14. 完成标准

1. Product Studio 的 Focus 与 Flow 在桌面/窄屏截图中均无半成品式空白、压缩或未样式化控件。
2. 工作台清楚展示设计规格、当前作品、UI 质量、视觉证据和 polish 结果。
3. 纯文字任务即使没有参考图，也必须通过桌面/移动 UI quality audit 才能成功。
4. 完整运行在首次全绿后执行一次有界 polish，并在全量复验后才交付。
5. SignalDesk 和 Atlas Research 均达到 3 页面、5 场景、桌面/移动证据和交付报告要求。
6. Atlas 的多张参考图真实进入 planner、页面映射、视觉门禁和修复证据。
7. README 主展示只保留成熟工作台和两个代表性演示；旧演示进入归档。
8. 所有既有安全、路径、依赖、秘密脱敏和成功门槛不回退。

## 15. 本轮不做

- 多 Agent 编排或模型主观评分。
- 任意技术栈、后端、数据库和真实第三方集成。
- 模板市场、团队协作、云端部署或多用户权限。
- 像素级复刻承诺或无界 polish/repair。
- 为移动端重新设计完整生产工作台。
