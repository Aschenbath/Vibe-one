# Vibe-one Product Lab 与截图生成设计

日期：2026-07-10

## 1. 目标

把 Vibe-one 从“能运行的 CLI/控制台”升级为适合真实使用和简历演示的 AI 产品生成工作台：

`文字描述和/或参考截图 → 多模态规划 → React Web 产品 → 构建与交互验证 → 视觉比较 → 有界修复 → 预览与交付报告`

本阶段继续只生成响应式 React + Vite Web 产品。原生 App、任意后端、多技术栈和远程托管不进入本轮；Java/Spring Boot 全栈模板是后续独立阶段。

## 2. 已确认的产品决策

- 视觉语言：明亮、清晰、有品牌感的 `Product Lab`，主色为高饱和蓝，辅以琥珀强调色和克制的成功绿。
- 信息架构：`Focus → Flow`。新建任务时专注输入；启动后切换为流水线与结果预览双区。
- 界面语言：所有面向用户的文案使用中文；模型名、API、原始事件代码等技术标识保留原文并作为次级信息显示。
- 使用目标：日常操作效率是骨架，简历展示所需的完成度、可解释性和证据链是外层表达，两者同时满足。
- 输入方式：文字描述与参考截图可以单独使用，也可以组合使用；不能把截图上传做成无效占位入口。
- 输出边界：当前只生成可在浏览器运行和自动验收的响应式 Web 产品。

## 3. 方案比较

### 方案 A：截图只作为模型提示

把参考图发送给 planner/builder，但不做机械视觉比较。实现快，但无法证明生成结果接近参考图，也不能可靠驱动视觉修复。拒绝。

### 方案 B：多模态规划 + 本地视觉比较 + 有界视觉修复

planner 从截图提取结构化视觉规格，builder 按规格生成；Playwright 产出结果截图，本地算法给出可复现的粗粒度相似度，失败时 fixer 同时看到参考图、结果图、指标和源码。选择此方案。

### 方案 C：多 Agent 视觉理解、DOM 重建和多技术栈生成

上限更高，但会引入复杂编排、模型自评和不可控输出，超出当前简历项目所需边界。延后。

## 4. Product Lab 工作区

### 4.1 新建状态

- 顶部只保留品牌、当前模型状态、运行设置和历史入口。
- 历史记录默认收起为窄栏；展开后以抽屉显示任务标题、状态、模型和时间。
- 主画布标题为“你想做什么产品？”。
- 文字区使用大尺寸、有明确标签的需求编辑器，不放营销式占位文案。
- 参考图区支持点击选择、拖放、缩略图预览、删除和顺序调整。
- “运行设置”收进侧边抽屉，包含执行模式、API endpoint、模型和会话级 API key；避免所有配置长期占据主画布。
- 主行动按钮为“开始生成”，旁边明确说明将执行规划、构建、验证和有界修复。

### 4.2 运行状态

- 页面从输入画布平滑切换为双区工作台，不进行整页跳转。
- 左区约 42%：阶段进度、当前动作、事件流、耗时和修复轮次。
- 右区约 58%：默认显示生成产品实时预览；同一区域切换参考图、结果截图、视觉比较、报告和修复记录。
- 阶段为：理解需求、生成产品、构建验证、视觉校验、自动修复、交付完成。
- 原始事件代码使用等宽小字显示，中文摘要作为主信息。
- 历史任务被选择后复用同一工作区，不重新出现永久三栏布局。

### 4.3 视觉与交互规范

- 页面背景使用冷白/浅灰，主表面使用纯白；蓝色只用于主行动、焦点和当前步骤。
- 不使用紫色渐变、Emoji 图标、厚重玻璃拟态或大面积黑色终端。
- 图标使用统一描边 SVG；交互区域不小于 44px。
- 动画使用 180–260ms，表达状态连续性；支持 `prefers-reduced-motion`。
- 明确覆盖 hover、focus-visible、pressed、disabled、loading、error、empty 和 success 状态。
- 桌面浏览器是主要使用环境；窄屏仍保持无横向溢出和可完成核心流程，但不为手机重新设计独立产品形态。

## 5. 截图输入契约

`POST /api/jobs` 继续使用 JSON，新增：

```json
{
  "title": "产品标题",
  "brief": "可为空，但 brief 与 references 至少提供一个",
  "references": [
    {
      "name": "home.png",
      "type": "image/png",
      "width": 390,
      "height": 844,
      "base64": "..."
    }
  ],
  "mode": "run",
  "baseUrl": "...",
  "model": "..."
}
```

约束：

- 支持 PNG、JPEG 和 WebP。
- 最多 4 张，每张解码后不超过 6 MiB，总计不超过 18 MiB。
- 服务器校验 base64、MIME、文件魔数、尺寸范围和安全文件名；不信任浏览器传来的扩展名。
- `/api/jobs` 使用 26 MiB JSON 请求上限以容纳 18 MiB 原图的 base64 膨胀；其他 API 继续使用现有小上限。
- 图片立即写入 jailed 的 `input/references/`，之后从内存任务对象丢弃 base64。
- API key、base64 和绝对路径不得进入事件、公开 job 对象、报告或日志。
- run 创建时把参考图复制到 `run/references/`，使历史任务仍可查看输入证据。
- job 公开对象只增加 `referenceCount`、安全文件名和输入模式，不返回磁盘路径。
- `loadConfig` 允许 brief 为空，但 brief 与 `input/references/` 至少存在一个；CLI 与浏览器控制台共用同一输入契约。

## 6. 多模态规划

provider 的 `chat`/`chatJson` 接受字符串或 OpenAI-compatible content 数组。存在参考图时，planner 的用户消息包含：

1. 文字需求；
2. 文件名和尺寸说明；
3. 每张图片的 `image_url` data URL。

planner 的严格 JSON 增加：

```json
{
  "visualDesign": {
    "layout": "结构描述",
    "palette": ["#..."],
    "typography": "字体层级与气质",
    "spacing": "密度与间距",
    "components": ["关键视觉组件"],
    "responsive": "响应式规则"
  },
  "pages": [
    {
      "name": "页面名",
      "route": "/",
      "referenceImage": "home.png"
    }
  ]
}
```

- 只有 planner 进行首次视觉理解，builder 使用已批准的结构化规格，避免重复发送大图和增加不确定性。
- 多张图由 planner 按文件名、顺序和内容映射到页面；没有对应参考图的页面只执行现有内容和交互检查。
- 网关不支持视觉输入时必须明确失败并在中文 UI 中说明，不允许静默忽略参考图后继续声称完成。

## 7. 机械视觉验证

### 7.1 截图尺寸

- 有参考图的页面优先使用其宽高作为 Playwright viewport，并限制在安全范围内。
- 无参考图页面继续使用配置中的默认 viewport。
- runner 继续保存现有 full-page 证据截图，同时为有参考映射的页面保存一张参考 viewport 尺寸的比较截图。
- 参考图与比较截图均归一化到固定采样尺寸，原文件和 full-page 证据截图保持不变。

### 7.2 相似度

新增纯本地比较器，不调用模型自评：

- 归一化亮度结构相似度（SSIM）权重 70%。
- RGB 颜色直方图相似度权重 30%。
- 综合得分范围为 0–1，默认通过阈值为 0.62，可由受控配置覆盖。
- 报告记录结构分、颜色分、综合分、阈值和对应文件名。
- 该指标明确称为“粗粒度视觉一致性”，不宣称像素级复刻质量。

比较器可通过 Playwright/Canvas 读取现有 PNG/JPEG/WebP，避免给模型生成的 App 增加依赖，也不改变固定依赖白名单。

### 7.3 Reviewer

reviewer 保持纯机械：

- 参考图映射到的每个页面必须产生视觉比较结果。
- 综合分低于阈值时 review 失败并进入现有有界 repair loop。
- brief-only 任务不新增视觉门槛，保持向后兼容。

## 8. 视觉修复

当视觉检查失败时，fixer 获得：

- 失败页面的参考图；
- 当前生成截图；
- 结构分、颜色分、综合分和阈值；
- 现有机械检查、交互失败和当前源码。

fixer 仍只返回完整文件内容，仍受 safeJoin、禁写清单、固定依赖和最大修复轮次约束。视觉失败不能扩大模型文件权限。

修复完成后重新执行构建、截图、交互和视觉比较；最终成功必须同时通过全部机械检查。

## 9. 证据与报告

- 证据区新增“参考图”和“视觉比较”视图。
- 每个比较卡展示参考图、结果图、综合分、子分和通过状态。
- Delivery Report 增加输入模式、参考图清单、视觉映射、每轮分数变化和最终结果。
- 公开文档只使用脱敏后的演示输入，不记录私人 endpoint 或 API key。

## 10. 错误处理

中文用户错误至少覆盖：格式不支持、图片过大、图片数量超限、文件损坏、缺少文字和截图、模型不支持视觉、请求体过大、视觉比较失败和修复轮次耗尽。

服务端内部 error code 保持稳定英文标识，前端按 code 映射中文消息；未知错误使用安全的通用提示，不回显上游响应中的秘密。

## 11. 测试策略

### 单元/集成测试

- 图片 base64、魔数、MIME、数量、单图大小和总大小验证。
- provider 正确发送多模态 content 数组，并保持 streaming/JSON fallback。
- planner 正确保存 `visualDesign` 和页面映射。
- 比较器对相同图高分、明显不同图低分，并在固定 fixture 上确定性输出。
- reviewer 在低于阈值时失败，brief-only 行为保持不变。
- fixer 的视觉请求包含参考图和结果图，但事件、job 和报告不包含 base64。
- jailed reference 路径和安全静态读取。

### 浏览器 E2E

- 全中文新建工作区可输入 brief、添加/删除参考图并启动任务。
- 启动后从 Focus 状态切换到 Flow 双区，实时阶段和证据可见。
- stub pipeline 验证一条视觉直接通过路径和一条视觉失败后修复路径。
- 桌面 1440×900 做主要视觉验收；390px 窄屏只验证无横向溢出和核心控件可达。
- 页面文本、事件、报告和 DOM 不出现 API key 或 base64。

### 回归验证

- 现有 offline suite、console E2E 和完整 stub pipeline 必须保持通过。
- `node --check` 覆盖所有改动的前端和服务端 JS。
- 重新生成桌面/窄屏 README 截图并人工检查裁切、重叠、焦点和空状态。

## 12. 完成标准

1. 用户可以只写描述、只上传截图或组合两者启动任务。
2. 截图真实进入 planner 的多模态请求，并产生结构化视觉规格。
3. 生成页面通过现有构建、内容、交互检查以及新增的机械视觉检查。
4. 视觉失败可触发携带参考图和结果图的有界修复，并在报告中展示分数变化。
5. 中文 Product Lab 工作区完整覆盖新建、运行、失败、修复、完成和历史查看状态。
6. 安全边界、固定脚手架、依赖白名单、路径 jail、秘密脱敏和成功门槛不回退。

## 13. 本轮不做

- 原生 iOS/Android 生成。
- 任意技术栈或任意后端生成。
- Java/Spring Boot 全栈模板。
- 多模型/多供应商编排。
- 多用户、远程访问、云端部署和并发任务。
- 像素级复刻承诺、模型主观打分或无界自动修复。
